## 1 解决的问题

### 1.1 背景与痛点

计算机辅助设计（CAD）是工程与制造领域最依赖专业技能的任务之一。现有的文本到CAD生成方法存在两类典型缺陷：

- **单次生成、无几何验证**（如 Text-to-CadQuery）：一次性输出结果，不验证生成几何体是否满足原始提示的尺寸与拓扑要求。
- **依赖有损视觉反馈**（如 Query2CAD）：通过图像编码提供视觉反馈，但编码表示丢失了几何精度，无法纠正毫米级的尺寸误差。

LLM 在生成 CAD 代码时极易产生幻觉尺寸和误用 API，且仅凭视觉反馈无法提供足够精确的纠正信号；而仅凭数值检查，又可能发生"伪收敛"——数值指标看似合格，但形状实际上与提示严重不符。

### 1.2 核心需求

需要一种反馈信号，既能提供**精确的数值几何度量**，又能提供**全局形状的高层感知**，两者协同工作才能可靠地生成满足工程精度要求的 CAD 模型。

## 2 方法

### 2.1 整体架构

  <img src={require("./images/1-CADCG_NID1_2-1CADSmith pipeline overview.png").default} style={{ width: "50%" }}/>
  <figcaption>图2-1 CADSmith pipeline overview</figcaption>
</figure>

CADSmith 是一个**多智能体流水线**，由五个专用智能体构成，通过两个嵌套修正循环驱动迭代优化：

- **内循环**（最多3次）：处理代码执行错误（Error Refiner）。
- **外循环**（最多5次）：处理几何验证失败（Refiner），以 OpenCASCADE 核心度量与 VLM 视觉判断为联合反馈信号。

完整流程：自然语言提示 → Planner → Coder（+RAG）→ Executor → Validator/Judge → Refiner → 最终 STEP/STL 输出。

### 2.2 各智能体职责

**Planner**：将自然语言提示转化为结构化设计方案（JSON格式），包括子部件列表、目标包围盒尺寸、几何约束（孔数、孔径、对称性）等，不生成代码。

**Coder**：接收设计方案，生成可执行的 CadQuery Python 脚本。借助两个检索知识库（RAG）增强：

- KB1（API文档）：155条 CadQuery Workplane 方法条目 + 28个代码示例，含方法签名、使用示例与常见陷阱。
- KB2（错误-解决方案模式库）：25条涵盖常见 CadQuery/OpenCASCADE 失败模式的条目，含触发关键词、根因解释与修复示例。

两个知识库均采用**关键词匹配**而非向量检索，在当前语料规模下具有确定性且无需维护嵌入模型。

**Executor**：在隔离的 Python 子进程中执行代码（60秒超时），成功则导出 STEP/STL 并从 OpenCASCADE 核心提取精确几何度量（体积、包围盒、质心、面/边/顶点计数、实体有效性）；失败则捕获完整错误追踪，触发内循环。

**Validator/Judge**：采用 **LLM-as-a-Judge** 架构，使用比代码生成更强的模型（Claude Opus vs Claude Sonnet）以避免自我确认偏差。Judge 接收四类输入：
1. 原始用户提示
2. 生成的 CadQuery 代码
3. Executor 提取的精确核心度量
4. 使用 VTK（Phong着色，2400×800分辨率）渲染的三视图（等轴测图、高角度后视图、正面轮廓图）

Judge 综合上述信息作出通过/失败决定并给出分析反馈；还强制执行硬性核心检查：若实体不合法（非水密），无论 Judge 评估结果如何均视为失败。若前次反馈指出的问题持续存在，Judge 将升级为建议采用完全不同的构造方案。

**Refiner**：接收当前代码、Judge 的反馈（含精确数值差异）、设计方案及原始提示，生成修正代码。接收完整历史记录以防止震荡；第3次迭代起获得明确的升级指引，促使其重新考虑整体构造策略而非反复调整同一参数。

### 2.3 评估基准

自建包含100条自然语言提示的基准数据集，分三个难度层次：

| 层次 | 描述 | 数量 | CadQuery操作数 |
|------|------|------|----------------|
| T1：基础图元 | 单一几何体（长方体、圆柱、圆锥、圆环等） | 50 | 1~3 |
| T2：工程零件 | 含布尔运算（支架、法兰、齿轮、孔阵列等） | 25 | 3~8 |
| T3：复杂零件 | 含工作平面变换、放样、扫掠、壳体等 | 25 | 5~15 |

所有参考脚本均由人工编写，执行验证并目视检查，以确保参考几何与提示意图严格一致。

### 2.4 评估指标

所有指标在**绝对毫米空间**（非归一化坐标）中计算，并在计算前通过 ICP 配准消除方向差异：

- **Chamfer Distance (CD)**：双向最近邻平均平方距离（10000个采样点），越低越好。
- **F1 Score**：在距离阈值 τ=1.0mm 下的精确率与召回率的调和平均。
- **体积 IoU**：1.0mm 分辨率体素化网格的交并比。

完整流水线将 CD 均值从 28.37 降至 0.74（降低 38 倍），执行成功率达到 100%。100条条目中 88 条在首次迭代即收敛，平均每条仅需 0.13 次精化迭代。

### 3.2 分层性能

T1 和 T2 层次实现接近完美的 F1（0.999 和 0.998），T3 复杂零件 F1 中位数为 0.886。视觉反馈在 T3 上的作用最为关键：移除视觉输入后，T3 的 CD 均值从 1.42 激增至 49.68（35 倍退化），F1 均值从 0.85 降至 0.74；而在 T1/T2 上，无视觉配置的表现与完整流水线相当，说明核心度量对简单零件已足够，但对复杂几何则不足以避免伪收敛。

### 3.3 典型失败模式

- **结构性失败**：T3_016（F1=0.57）、T3_024（F1=0.59）——生成方案与参考方案在构造策略上差异显著，产生几何上合法但结构上不同的实体。
- **近似未通过（Near-miss）**：T3_019 四轴机架（F1=0.963，IoU=0.985）——各项指标与 Judge 视觉评估均通过，但机臂与中心毂之间存在微小间隙，难以在渲染视图中发现，实际上无法用于制造。

## 4 优势与不足

### 4.1 优势

- **精确几何反馈**：基于 OpenCASCADE 核心的程序化度量提供精确数值信号，而非模糊的视觉编码。
- **双重反馈互补**：数值核心度量 + VLM 视觉判断协同工作，两类信号互相弥补对方的盲区（伪收敛 vs 无法精确定位误差）。
- **无需微调**：采用 RAG 而非 fine-tuning，随 CadQuery API 演进仍保持时效性。
- **避免自我确认偏差**：使用更强的独立模型（Claude Opus）作为 Judge，不同于生成代码的 Claude Sonnet。
- **性能显著提升**：CD 均值相较 zero-shot 基线降低 38 倍，执行成功率达 100%。

### 4.2 不足

- **三视图覆盖不足**：固定的三个视角无法检测小尺度几何错误（如接合处间隙、不完整布尔运算），如 T3_019 案例所示。
- **基准规模有限**：100条自建基准，难以全面覆盖制造相关几何的多样性。
- **多零件装配未支持**：当前仅支持单体零件生成。
- **单一模型族评估**：仅在 Claude 系列上评估，未跨多种前沿模型比较。

## 5 创新点

- **多智能体分解架构**：将文本到CAD生成分解为 Planner / Coder / Executor / Validator / Refiner 五个专用智能体，取代单体式单模型方案。
- **双循环程序化几何验证**：内循环修正执行错误，外循环以 OpenCASCADE 核心度量驱动迭代几何纠正，首次将程序化验证闭环引入文本到CAD系统。
- **独立 VLM-as-Judge**：将独立的强视觉语言模型（Claude Opus）嵌入修正循环内部用作反馈，而非仅用于事后评估，有效避免自我确认偏差。
- **基于 RAG 的 API 文档检索**：无需微调，通过关键词匹配检索 API 文档与错误模式库，保持对 CAD 库版本迭代的适应能力。
- **绝对毫米空间评估**：采用绝对坐标而非归一化坐标计算度量，同时捕获形状准确性与尺寸准确性。

## 6 个人感悟

CADSmith 的核心思路在于将 LLM 自精化（Self-Refine）范式与工程领域的硬性约束相结合——CAD 不同于图像生成，"看起来对"远远不够，毫米级的尺寸误差就意味着零件报废。因此，单纯的视觉反馈或单纯的数值反馈均不充分，必须将两者互补地整合进同一个闭合循环。

使用更强的独立模型作为 Judge 这一设计选择值得关注：它不仅回避了自我确认偏差，实际上也反映了工程评审流程中"代码审查"与"设计审查"分离的工程实践智慧。

T3_019 的近似失败案例（指标全部通过但存在制造级缺陷）揭示了当前基于固定视角渲染反馈的根本局限性，自适应视角选择（如聚焦于接合部或几何复杂区域）是一个值得深入的研究方向。

此外，RAG 替代微调的策略具有较强的工程实用性：避免了昂贵的数据收集与训练开销，同时能跟随 API 版本迭代保持时效性，是在快速演进的工具生态下构建稳健系统的一种值得借鉴的思路。
