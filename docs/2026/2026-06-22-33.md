## 1 论文概览

### 1.1 一句话总结

TOOLCAD 提出让大语言模型（LLM）作为"工具调用型智能体"，通过 Model Context Protocol（MCP）直接操控 CAD 引擎（FreeCAD）完成文本到 CAD 建模，并配套设计了交互式 CAD 建模 Gym（混合反馈）与"SFT + 在线课程式 GRPO"两阶段后训练策略，使开源 7B/8B 级模型在建模成功率和几何精度上超过未经训练、仅靠提示工程的 GPT-4o、Qwen3-235B 等更大规模模型。

### 1.2 研究背景与动机

CAD（计算机辅助设计）是工业产品全生命周期中的核心工具，但专家级建模耗时且依赖人工几何理解能力。现有 LLM/VLM 驱动的 CAD 生成方法大体分两类：

1. **LLM 辅助参数化 CAD 序列生成**：把 LLM 当作辅助模块做 token 级预测（如 Text2CAD、CAD-GPT、CAD-MLLM）；
2. **LLM 生成/修正 CAD 代码**：利用 LLM 生成或迭代修正代码级建模指令（如 CAD-Assistant、CAD-Llama、CADCodeVerify、Seek-CAD）。

作者认为这两类方法都没有解决"LLM 如何与 CAD 引擎进行最优的多轮交互"这一问题，这正是 TOOLCAD 想填补的空白（即论文反复强调的 **Prompt-to-Tool** vs **Prompt-to-Code** 的区别，详见原文[图1](#figure1)）。

具体而言，论文指出了三个核心挑战：

1. LLM 本身的推理能力和工具集成能力有限，难以支撑长程、多步骤的 CAD 建模决策；
2. 缺乏带建模反馈的交互式 CAD 环境，也缺乏针对"工具调用型"CAD 智能体的评测基准；
3. 面对复杂程度不同的文本到 CAD 任务，现有训练方式（如纯 SFT）训练不充分，泛化能力差。

## 2 核心方法：TOOLCAD 框架

### 2.1 问题建模：CAD 建模作为 MDP

作者将"工具调用型智能体执行 CAD 操作"的过程建模为一个有限时域的马尔可夫决策过程 $\mathcal{M} = \{S, A, T\}$：状态 $s$ 是历史推理与之前建模动作的上下文；给定专家级指令 $I$，策略 $\pi_\theta$ 在每一步根据当前状态 $s_t$ 选择一个工具动作 $a_t$，并从 CAD 建模 Gym 获得反馈 $(r_t, s_{t+1})$，从而形成可用于训练的"推理-工具调用-反馈"轨迹：

$$
a_t \sim \pi_\theta(\cdot \mid s_t, \tau_{<t}, I), \quad (r_t, s_{t+1}) \sim \mathcal{P}(\cdot \mid s_t, a_t)
$$

其中 $\tau_{<t} = \{s_0, a_0, r_0, ..., s_{t-1}, a_{t-1}, r_{t-1}\}$ 表示包含全部前序推理引导工具调用、观测和奖励的历史交互。

### 2.2 框架三阶段

TOOLCAD 工作流（原文[图2](#figure2)）包含三个阶段：

#### （1）CAD 建模决策（Decision-making）

利用 Chain-of-Thought 提示和后训练，让 LLM 针对 L3 专家级指令生成"CAD 建模思维链"（CAD-CoT），完成任务规划与拆解。L3 专家级指令包含密集数值参数（坐标、角度、半径等），覆盖坐标系设置、草图（Sketching）、拉伸（Extrusion）、布尔运算（cut/union/common）的标准化、序贯的建模流程。模型被要求按照严格格式输出：`<think>...</think>` → `<tool_call>...</tool_call>` → `<tool_response>...</tool_response>`。

#### （2）配备建模工具与环境

智能体依据 CAD-CoT 中的推理引导动作，通过自建的 MCP（Model Context Protocol）工具池调用 FreeCAD 中的建模原语，与 CAD 环境交互。论文实现的工具库包括：

```
freecad-set_coord_system
freecad-create_complex_sketch
freecad-create_simple_sketch
freecad-boolean_operation
freecad-multiple_fuse
freecad-extrude_face
```

MCP Server/Client 的集成架构如原文[图11](#figure11)所示，整体上是"设计者输入指令 → LLM 选择并请求 MCP 工具 → FreeCAD MCP-Server 执行 → 返回工具输出/观测 → LLM 生成反思"的闭环。

#### （3）反思式自动建模（Reflective ReAct）

采用 CAD 专用的 ReAct 范式，基于引擎返回的混合反馈（成功/失败标签 + 错误信息 + 当前几何对象列表）反思每次工具调用结果，迭代修正直到任务成功或失败。

### 2.3 CAD 建模 Gym：混合反馈机制

论文设计了两层反馈：

- **自发反馈（Spontaneous Feedback）**：暴露 CAD 引擎原始的几何冲突、约束警告、API 报错信息（粒度较粗、夹杂代码片段），帮助智能体回溯纠错。例如 `create_complex_sketch` 工具失败时会返回"建议逐个轮廓环创建"的纠错提示（详见原文[图12](#figure12)）；`boolean_operation` 失败时返回带操作类型、对象名和错误信息的反馈（详见原文[图13](#figure13)）。
- **人工增强反馈（Human-Augmented Feedback）**：把引擎原始反馈重新包装成"success / fail"的结构化自然语言描述，并维护一份当前几何对象列表（而非仅依赖历史交互记录），用以减少工具调用幻觉，支持更可靠的反思式建模。

此外，论文构建了一条**示范数据采集流水线**（原文[图3](#figure3)上半部分）：用多个前沿 LLM（GPT-4o、Qwen3-235B-A22B、Gemini-1.5-pro、Qwen3-32B）做推理生成轨迹，经人工视觉核查与目标几何对比，对错误轨迹给出修正指令进行二次生成，最终收集 982 条成功的建模工具调用轨迹（按 part 数量 1/2/3/4/≥5 分层采样，得到 782 条训练轨迹 + 200 条测试轨迹，详见[表1](#table1)）。

## 3 奖励建模与两阶段训练策略

### 3.1 混合奖励设计

#### （1）结果监督奖励模型 ORM（$\mathcal{M}_{\text{ORM}}$）

用人工标注的成功/失败示范数据（含负样本）微调一个 LLM 作为奖励模型，输入指令 $I$ 与完整工具调用轨迹 $\tau$，输出 YES/NO 判断任务是否成功完成目标几何；在线 RL 阶段，若 ORM 判定"YES"概率更高则给奖励 1，否则给 0。

#### （2）规则奖励（Rule-Based Reward）

- **步骤级执行奖励**：从每次工具调用后的反馈中提取二元信号（"success"或"fail"），公式为：

$$
\mathcal{R}(s_t, \mathcal{O}_t) = \text{EM}(\mathcal{O}_t, \text{label})
$$

- **格式奖励**：检查模型输出是否严格遵循 `<think>` → `<tool_call>` → `<tool_response>` 的顺序与完整性，符合则额外给予 0.5 的奖励。

#### （3）总奖励

总奖励是上述三者的加权组合：

$$
R(t_i) = \alpha \cdot R_{\text{ORM}} + \beta \cdot \left(\frac{1}{T}\sum_{t=1}^{T} R_{\text{step}}^{(t)}\right) + \gamma \cdot R_{\text{format}}
$$

### 3.2 两阶段后训练流程

整体训练流程如原文[图3](#figure3)所示：

1. **Step Ⅰ — SFT & 离线 RL**：用收集到的成功示范轨迹做行为克隆（Behavioral Cloning，本质等价 SFT），目标是最大化：

$$
J_{\text{BC}}(\theta) = \mathbb{E}_{\tau \sim D} \sum_{t=1}^{T} \left[ \log \pi_\theta(a_t \mid s_{t-1}) \right]
$$

由此得到初始策略 $\mathcal{M}_{\text{SFT}}$（也对比了 AWR 作为离线 RL 基线）。

2. **Step Ⅱ — 在线课程式 RL（Online Curriculum RL）**：在 held-out 任务上用 GRPO 持续优化策略，并配合"分部件课程学习策略"逐步增加任务复杂度（part 数）。

### 3.3 Part-Wise 课程学习策略（关键创新点）

CAD 模型中的部件数（unit 数）直接决定建模复杂度和轨迹长度，过长轨迹容易导致上下文溢出和灾难性遗忘。论文设计了基于**动作序列平均困惑度**的课程策略：

$$
\mathcal{P}(\tau) = \exp\left(-\frac{1}{T}\sum_{t=1}^{T}\log \pi_\theta(a_t \mid s_t)\right)
$$

对当前难度等级，先计算 held-in 验证集上的初始困惑度阈值 $\delta'$，再将实际阈值软化为 $\delta = \alpha \cdot \delta'$（$\alpha \in (0,1)$ 控制探索-利用权衡）；当策略在当前难度上的平均困惑度低于 $\delta$ 时，才进入下一课程阶段（part 数更多的任务）。完整算法见原文 Algorithm 1。

### 3.4 GRPO 训练目标

采用 Group Relative Policy Optimization（GRPO）做 critic-free 的轨迹级优化：

$$
J_{\text{GRPO}}(\theta) = \frac{1}{G}\sum_{i=1}^{G}\frac{1}{|\tau_i|}\sum_{k=1}^{|\tau_i|}\min\left[\frac{\pi_\theta}{\pi_{\text{old}}}\hat{A}_{i,k},\ \text{clip}\left(\frac{\pi_\theta}{\pi_{\text{old}}}, 1-\varepsilon, 1+\varepsilon\right)\hat{A}_{i,k}\right] - \beta D_{\text{KL}}[\pi_\theta \| \pi_{\text{ref}}]
$$

结合 PPO 风格的 clip 机制和相对参考策略 $\pi_{\text{ref}}$ 的 KL 正则项保证训练稳定性。训练细节：基于 Verl 和 LLaMA-Factory 框架，8×H20-96GB GPU；SFT 学习率 $1e{-5}$（余弦调度，0.1 warm-up）；在线 RL 阶段每任务采样 8 条 rollout，温度 0.1，生成预算 8192 token，最大上下文 16384 token，GRPO batch size 32，学习率 $2e{-6}$。

## 4 主要实验结果

### 4.1 数据集与评测指标

基于 DeepCAD（约 67K 模型）与 Text2CAD（约 170K 多级标注，L0~L3）数据集，选用 L3 专家级指令作为主任务指令。DeepCAD 的主要局限是多部件任务稀缺（多数模型仅含 1 个 part），因此作者额外构造了上文提到的 982 条示范轨迹用于 SFT 初始化和 ORM 训练。

主要评测指标包括：Invalidity Ratio（IR，无效模型占比）、Chamfer Distance（CD）、Minimum Matching Distance（MMD）、Intersection over Union（IoU / $\text{IoU}_{\text{best}}$）、Parameter Density（参数密度）、Coverage（COV，分布覆盖度）、Jensen-Shannon Divergence（JSD，体素分布差异）、平均建模成功率（Avg.SR）、工具调用准确率、Sketch/Extrusion F1 等。

### 4.6 指令层级泛化能力

即便在更模糊的 L0/L1 指令下，TOOLCAD 依然显著优于 Text2CAD（如 L0 下 COV 58.19% vs 40.27%，Novelty 98.14% vs 94.33%），说明其学到的建模范式具备一定的"自主补全意图"能力，不完全依赖详细的 L3 参数化描述。

### 4.7 ORM 评测能力与案例研究

自训练的 ORM 在测试用例和困难用例（≥5 部件）上的判别准确率（82.7%/65.2%）显著超过用 GPT-4o、Qwen3-235B、Qwen2.5-VL 直接做裁判的方案（最低 Qwen2.5-VL 仅 55.3%/41.6%）。原文[图5](#figure5)的案例研究显示：ORM（100% 训练）能检测出"缺失部件单元""布尔运算错误"等视觉上与 Ground Truth 几乎无法区分的细微失败，而 VLM 裁判（Qwen2.5-VL）对这些 near-miss 失败误判为成功。

### 4.8 消融实验

原文[图6](#figure6)、[图7](#figure7)、[图8](#figure8)的消融结果：

- **去掉步骤级规则奖励（w/o StepRW）**：性能明显退化，说明仅靠稀疏的 ORM 终局奖励不足以支撑有效的步骤级优化。
- **去掉课程学习（w/o CL）**：相比完整版收敛更慢、方差更高，在 held-out 任务数增加时缺乏结构化难度引导。
- **课程阈值系数 $\alpha$ 的影响**：$\alpha=0.9$（探索浅）智能体快速进入高难度阶段但困惑度居高不降，技能学习不足；$\alpha=0.5$（探索深）出现困惑度剧烈震荡，训练不稳定；$\alpha=0.7$ 时困惑度平稳下降，是较好的探索-利用平衡点。

### 4.9 效率与规模分析

- **效率对比**（原论文 Table 8）：TOOLCAD(Qwen2.5-7B) 每次工具调用平均 5178 token、648ms 延迟，显著低于 GPT-4o（11703 token/982ms）和 CAD-Assistant(GPT-4o)（30802 token/1326ms），体现工具调用型小模型在效率上的优势。
- **规模法则**（原文[图9](#figure9)）：SFT-only 基线模型参数从 0.5B 增至 3B 时有明显提升，但超过 7B 后进入性能平台期；而在线 RL 数据量从 1× 增加到 5× 带来的提升远大于单纯增加模型参数，说明在当前 RL 范式下，"RL 训练数据规模"是比"模型规模"更关键的性能驱动因素，但 RL 训练的不稳定性会限制这种规模化收益的上限。

## 5 内容要素总结

### 5.1 解决何问题

如何让大语言模型在长程、多步骤、参数密集的文本到 CAD 建模任务中，以"工具调用智能体"的方式直接、可靠地操控真实 CAD 引擎完成建模，而非依赖纯 token 级序列预测或纯代码生成；同时解决该方向上"缺乏交互式训练环境"和"缺乏有效后训练策略"两个配套难题。

### 5.2 大致采取何方法

1. 用 MCP 协议把 FreeCAD 的建模原语封装为标准化工具，让 LLM 以 CAD 专用 ReAct（CAD-CoT）格式进行"推理→工具调用→反馈→反思"的多轮交互；
2. 搭建交互式 CAD 建模 Gym，提供"自发反馈（引擎原始报错）+ 人工增强反馈（结构化成功/失败描述）"的混合反馈；
3. 通过人机协同的示范数据采集流水线收集成功轨迹，先做 SFT/行为克隆得到初始策略；
4. 训练结果监督奖励模型 ORM，结合步骤级规则奖励和格式奖励构成混合奖励函数；
5. 用基于动作序列困惑度的"分部件课程学习"策略，配合 GRPO 做在线强化学习，逐步提升任务复杂度，演化出鲁棒的 CAD 工具调用智能体。

### 5.3 优势与不足

**优势：**

- 相比直接生成 CAD 代码/序列，工具调用方式天然获得引擎级的"合法性校验"，显著降低无效模型率（IR）和几何误差（MCD）；
- 用 ORM/规则奖励等程序化、符号化反馈替代 VLM 视觉判别，在检测"缺失部件""布尔运算错误"等细粒度失败上更可靠，避免了视觉幻觉问题；
- 课程学习策略有效缓解了长轨迹导致的上下文溢出和训练不稳定问题；
- 7B/8B 级开源模型经训练后可超越未训练的更大规模模型（GPT-4o、Qwen3-235B），且推理效率（token 消耗、延迟）更优，具备较好的实用性价值。

**不足（含作者自述局限）：**

- 完全依赖文本提示驱动工具调用，尚未引入视觉线索引导的上游感知模块；论文自己的 ToolCAD-VLM 变体效果反而不如纯文本方案，说明视觉信息的有效融合仍是开放问题；
- CAD 专用工具学习的探索有限：缺乏自我修正能力、ORM 性能存在退化风险、工具库规模有限；
- 数据规模偏小：982 条人工核验示范轨迹相对 DeepCAD ~67K 模型规模而言较小，高复杂度（≥5 part）样本仅 122+40 条，统计显著性有待更大规模验证；
- ORM 的失效模式（如被策略"reward hack"的风险）未充分讨论；
- "RL 数据规模比模型规模更关键"的结论可能仅在当前 GRPO 实现和中小模型规模区间内成立，泛化边界不明确。

### 5.4 创新点

- **Prompt-to-Tool 范式**：首次系统性探索 LLM 作为工具调用智能体直接操控 CAD 引擎（区别于 Prompt-to-Code 及 token 级序列预测）；
- **交互式 CAD 建模 Gym**：自发反馈 + 人工增强反馈的混合反馈机制，配合 MCP 协议标准化工具接口；
- **混合奖励体系**：ORM（结果监督）+ 步骤级规则奖励 + 格式奖励三者加权组合；
- **基于困惑度的分部件课程学习策略**：用策略自身的困惑度作为难度切换的自适应信号，而非依赖人工设定的难度标签。

### 5.5 个人感悟

TOOLCAD 的核心洞见在于：把领域专业知识（CAD 建模原语、几何约束、合法性校验）编码进工具接口和奖励函数中，让强化学习的探索聚焦在"何时调用何种工具、以何种参数调用"这一更受限、更可验证的空间，而非要求模型从零生成完整的代码或序列。这种"领域工具化 + 通用 RL 算法"的应用范式，与 Search-R1、AgentGym 等近期工具增强型 LLM Agent RL 工作思路一致，对其他需要与专业软件/引擎交互的智能体任务（如 EDA 设计、科学计算、机器人控制）具有方法论参考价值。

同时，论文反复用多组实验证明"文本/结构化反馈在捕捉细粒度建模错误上系统性优于视觉判别"，这提示在需要高精度结构判定的智能体任务中，符号化反馈可能比端到端视觉理解更可靠——这是一个值得在其他多模态 Agent 任务中验证和借鉴的结论。但论文对 ORM 自身可靠性边界、reward hacking 风险等问题讨论较浅，且示范数据规模偏小，距离真正稳健的工业级落地还有一定距离，后续工作可在更大规模、更多样几何分布、更强自我修正机制等方向继续推进。
