## 1 研究背景与问题定义

### 1.1 问题背景

参数化计算机辅助设计(Parametric CAD)是现代产品研发的基石。近年来基于 LLM 的方法已能从文本或图像生成<strong>单个参数化零件</strong>,且保真度尚可。然而,现实中的产品大多是<strong>装配体(assembly)</strong>:由多个部件通过带有规定自由度(Degrees of Freedom,DOF)的"有类型关节"连接而成。

本文聚焦的核心问题是:<strong>如何从高层描述(文本或图像)直接生成可编辑的、带铰接(articulated,即可运动)的 CAD 装配体</strong>。这一任务此前几乎无人探索。

### 1.2 现有方法的局限

作者将相关工作划分为两个"半边",指出它们各自只解决了问题的一部分:

1. <strong>CAD 代码生成方法</strong>(如 DeepCAD、Text2CAD、CAD-Recode、CAD-Coder 等):只能产出单个静态零件,缺乏关节机制与多部件层级结构。即便有 CADCodeVerify、Seek-CAD、EvoCAD 等"生成—执行—修复"循环框架,也仍局限于单一静态部件。

2. <strong>铰接物体重建方法</strong>(如 Ditto、PARIS、ArtGS、Articulate-Anything 等):从图像或视频推断几何与关节参数,但输出的是<strong>不可编辑的网格或 3D 高斯表示</strong>;泛化能力受限于训练时见过的物体类别(主要是家具),且缺乏供下游工程迭代所需的<strong>构造历史(construction history)</strong>。

因此,在"单部件 CAD 生成"与"不可编辑的 3D 重建"之间,存在一道明显的空白地带——而 ArtiCAD 正是为填补这道空白而提出的。

### 1.3 核心挑战

作者识别出两个关键挑战:

第一,<strong>任务由多个耦合的子问题构成</strong>:需求分析、结构设计、零件生成与最终装配。单个 LLM/VLM 难以可靠地一并处理,因此采用多智能体分工。

第二,也是最关键的,<strong>预测部件如何连接是本任务相较单部件生成所特有的难点</strong>。当前 LLM/VLM 的精确 3D 空间推理能力很弱。如果把连接关系的预测推迟到装配阶段——即让模型去观察各自独立生成的零件、再去猜测它们的坐标系与表面如何对齐——系统将面临巨大的搜索空间,并经常失败。

## 2 核心思想与方法

### 2.1 整体框架概述

ArtiCAD 是<strong>首个免训练(training-free)的多智能体系统</strong>,从文本/图像生成可编辑的铰接式 CAD 装配体。它模拟了"设计—生产—装配"的工作流,由四个专职智能体组成:

1. <strong>Design Agent(设计智能体)</strong>:接收用户输入(文本、图像或两者),输出包含零件规格与装配关系(即"连接器")的结构化方案;
2. <strong>Generation Agents(生成智能体)</strong>:依据设计方案,把每个零件独立生成为 FreeCAD 的 Python 脚本;
3. <strong>Assembly Agent(装配智能体)</strong>:利用 FreeCAD 的约束求解器对齐匹配的连接器对,构建最终模型;
4. <strong>Review Agent(评审智能体)</strong>:通过检查各零件的多视角渲染图与装配体关节运动的关键帧序列来评估结果。

此外,系统从两方面提升质量:其一是在生成与装配阶段引入<strong>验证步骤</strong>,配合<strong>跨阶段回滚机制</strong>;其二是<strong>自演化经验库</strong>,在每个任务后积累设计知识以提升后续表现。

### 2.2 铰接式装配的表示

#### (1)CAD 后端与几何表示

ArtiCAD 选用开源参数化平台 <strong>FreeCAD</strong> 实现装配。作者解释了选型理由:已有的代码式 CAD 生成方法多面向 CadQuery 或 OpenSCAD,但 CadQuery 只能做静态摆放、没有关节类型与运动学求解,OpenSCAD 则是纯 CSG、没有装配概念;而 FreeCAD 兼具基于草图的特征建模(pad、pocket、revolve、loft)与基于约束的装配求解器。

一段生成的 Python 脚本 $\mathcal{G}_i$ 经底层 OpenCASCADE 内核求值后,得到边界表示(B-rep)实体,该实体由面、边、顶点等拓扑实体 $\mathcal{T}_i$ 构成。但作者指出,跨零件直接引用这些拓扑实体并不稳定(见 2.3 的拓扑命名问题),因此其表示将连接接口<strong>抽象为显式的局部坐标系</strong> $\mathbf{c} \in SE(3)$,而非隐式的拓扑面。

#### (2)运动学关节

ArtiCAD 不把关节定义为"动态拓扑表面之间的重合约束",而是定义为<strong>两个显式局部坐标系之间的运动学约束</strong>。实现中使用五种核心关节类型,均可通过 FreeCAD 的 Python API 在无头(headless)模式下脚本化调用,如[表2-1](#table2-1)所示。给定这些有类型约束后,求解器即可计算出满足约束的确定性 $SE(3)$ 刚体变换,从而把装配与底层 B-rep 拓扑完全解耦。

#### (3)从装配图到运动学树

铰接装配天然构成一张以关节为边的图,但<strong>含闭环的图会带来耦合约束</strong>,LLM 很难保持其一致性。因此作者把结构限制为<strong>运动学树(kinematic tree)</strong> $\mathcal{T} = (\mathcal{P}, \mathcal{J}, g)$——这也是机器人学与具身 AI(如 URDF)中的标准铰接表示。其中 $N$ 个参数化零件 $\mathcal{P} = \{p_i\}_{i=1}^{N}$ 由 $N-1$ 个有类型关节 $\mathcal{J} = \{j_k\}_{k=1}^{N-1}$ 连接,并以一个固定于世界坐标系的接地零件 $g \in \mathcal{P}$ 为根。每个关节携带一个来自上述五种类型的 $\tau_k$ 及可选的运动限位。总自由度为:

$$
D = \sum_{k=1}^{N-1} \delta(\tau_k), \quad \delta(\tau) = \begin{cases} 0 & \tau = \texttt{Fixed} \\ 1 & \tau \in \{\texttt{Revolute}, \texttt{Slider}\} \\ 2 & \tau = \texttt{Cylindrical} \\ 3 & \tau = \texttt{Ball} \end{cases}
$$

无环拓扑保证每个零件都有唯一父节点,从而求解器总能收到一个适定(well-posed)的问题。

### 2.3 连接器契约(Connector Contract)

#### (1)定义

<strong>连接器(connector)是本文最核心的设计</strong>:它是零件上一个具名的附着点,实现为带语义标签的局部坐标系:

$$
c = (n,\; \mathbf{o} \in \mathbb{R}^3,\; \hat{\mathbf{z}} \in \mathbb{S}^2,\; \hat{\mathbf{x}} \in \mathbb{S}^2,\; l)
$$

其中 $n$ 是唯一标识名,$\mathbf{o}$ 是零件局部坐标系下的原点,$\hat{\mathbf{z}}$ 是主轴(旋转或滑动方向),$\hat{\mathbf{x}}$ 是正交参考轴,$l$ 描述该附着点的语义用途。运动学树中每个关节都在它连接的两个零件上各引用一个连接器。

连接器扮演<strong>跨阶段契约(cross-stage contract)</strong>的角色:Design Agent 在规划期指定它们,Generation Agents 在已构造的实体上实现对应坐标系,Assembly Agent 则在<strong>无任何 LLM 调用</strong>的情况下对齐匹配对。一旦连接器被提前固定,装配就退化为确定性的坐标系对齐,且失败的零件可以被孤立地重新生成。

#### (2)早预测 vs. 晚预测

本文的关键洞见是:<strong>在设计阶段而非装配阶段完成装配关系预测</strong>。作者用"按图施工"而非"盲目拼拼图"的类比来说明——早预测在任何 3D 形状生成之前就先确立好连接计划,把"连接规划"与"形状生成"分离开,从而绕开 LLM 在最终装配时的空间推理短板。

作者还给出了形式化论证。设装配图有 $N$ 个零件和 $E$ 个关节:

- <strong>晚预测</strong>需要 LLM 跨零件匹配生成的拓扑实体(如 B-rep 面),搜索空间为 $O(V^{2|E|})$($V$ 为每个零件的平均拓扑特征数)。更糟的是,参数化 CAD 系统普遍存在<strong>拓扑命名问题(Topological Naming Problem,TNP)</strong>:微小的代码改动会不可预测地重新索引实体,使跨零件映射极不稳定。
- <strong>早预测</strong>通过预先建立连接器契约集合 $\mathcal{C} = \{c_i\}_{i=1}^{N}$,把上述搜索空间坍缩为 $O(1)$ 的确定性坐标系对齐。

从概率角度看,连接器契约相当于一个<strong>马尔可夫毯(Markov blanket)</strong>。相比"单一错误即触发全局级联失败"的整体联合分布(期望重试次数随 $O(K^N)$ 增长,$K$ 为每个零件的期望重试数),各零件的生成在给定契约后变为<strong>条件独立</strong>:

$$
P(\mathcal{G}_1, \dots, \mathcal{G}_N \mid \mathcal{C}) = \prod_{i=1}^N P(\mathcal{G}_i \mid \mathbf{c}_i)
$$

这种数学上的解耦把失败隔离开来,使期望回滚代价线性地受限为 $O(N \cdot K)$。

### 2.4 四个智能体

#### (1)Design Agent

Design Agent 把多模态输入转化为结构化方案,包含两步:

<strong>需求分析</strong>:一个基于 VLM 的模块把文本与参考图解析为组件、空间关系与约束的规格。对于欠规约(under-specified)的输入,一个<strong>头脑风暴(brainstorm)模块</strong>会提出若干结构上彼此不同的备选方案,供用户选择。

<strong>方案生成</strong>:结合规格与从经验库检索到的相似历史方案,输出一份声明式方案:

$$
\mathcal{P} = \bigl(\{p_i\}_{i=1}^{N},\; \{j_k\}_{k=1}^{N-1},\; g,\; D\bigr)
$$

其中每个组件 $p_i$ 携带设计参数、朝向提示与参考连接器;每个关节配对两个连接器并指定类型与限位;$D$ 为声明的总自由度。方案会被做<strong>结构性验证</strong>:关节图必须构成以 $g$ 为根的树,且 $D$ 必须与公式(1)一致。

<strong>派生(Derive)机制</strong>:对称或重复零件(如两扇镜像的冰箱门、四条作刚性平移的桌腿)只指定其一为"源",并为每个副本指定确定性的 $SE(3)$ 变换,这样只有"源"零件进入代码生成。

#### (2)Generation Agents

为方案中每个非派生组件各派生一个 Generation Agent:

<strong>几何构造</strong>:智能体生成 FreeCAD Python 脚本来建模零件几何,脚本在沙箱化的 FreeCAD 进程中执行;失败时,错误轨迹与部分几何的渲染图反馈回智能体,经"生成—执行—修复"循环修正。派生零件完全绕过 LLM——由确定性脚本对源几何及其连接器直接施加规划好的 $SE(3)$ 变换。

<strong>连接器实现</strong>:连接器契约充当<strong>语义化的空间约束</strong>,而非盲目地强制固定坐标。代码生成期间,智能体会根据本地生成几何的实际形状与尺寸,微调导出连接器坐标系的精确位置(例如恰好落在某圆柱顶面的圆心),从而在不破坏全局契约的前提下保证语义有效的定位。

<strong>局部验证</strong>:执行成功后,一个 VLM 把生成零件的多视角渲染图与初始规格比对,检查形状、比例与朝向;若发现不匹配,反馈被送往中央错误处理器(见 2.5)。

#### (3)Assembly Agent

Assembly Agent 用已实现的组件合成完整的运动学树:

<strong>确定性装配</strong>:给定生成的零件及其导出的连接器坐标系,确定性脚本计算刚体变换以对齐每个关节的连接器对,再施加 FreeCAD 装配约束建立关节。由于拓扑早已由 Design Agent 解决,这一物理对齐步骤<strong>不涉及任何 LLM 或 VLM</strong>。

<strong>全局验证</strong>:装配体经一条 VLM-LLM 流水线严格审查——VLM 先分析多视角渲染图与运动关键帧的空间与运动学有效性,LLM 评判者再结合包围盒数据与需求,就摆放、干涉与运动保真度给出结构化结论。否定结论会触发中央错误处理器。

#### (4)Review Agent 与记忆系统

ArtiCAD 通过一套由 Review Agent 主导的演化式记忆系统来缓解历史性错误与 API 幻觉:

<strong>Review Agent</strong>:装配后,对输出做 VLM 与规则结合的几何保真度与运动学健康度评估,并把完整的生成轨迹(需求、连接器方案、代码、修复轨迹)提炼为结构化案例摘要。

<strong>经验库(Experience Store)</strong>:摘要在 FAISS 中被划分为"Good"与"Issue"两类,支持<strong>非对称检索策略</strong>:Design Agent 同时从两类分区提取正向设计启发与负向约束,而 Generation Agent 严格只把 Good Cases 当作干净的少样本模板。该循环无需微调即可提升成功率。

<strong>文档库(Documentation Store)</strong>:为弥合用户意图与 CAD API 命名之间的语义鸿沟,采用意图驱动检索——由 LLM 从几何预测可能的 API 签名,再用其嵌入去查询切块后的文档,为各智能体提供精确语法。

### 2.5 跨阶段回滚机制

在生成与装配阶段引入 VLM 验证的首要动机,是让它们充当<strong>分布式传感器</strong>。为了让这些多阶段反馈可被付诸行动而又不丢弃成功的中间产物,ArtiCAD 采用跨阶段回滚机制(即图中的 Error Handler、Classifier 与 Router):

<strong>错误分类与路由</strong>:当任一 VLM/LLM 评判者报告失败时,路由器分析反馈以定位缺陷,并将其分类为两种之一——<strong>CODE 错误</strong>(Python 脚本未能实现一个有效规格,如孔径尺寸错误)或 <strong>DESIGN 错误</strong>(底层连接器方案在逻辑或物理上有缺陷,如装配后部件严重干涉)。据此把具体的视觉诊断与错误轨迹回路到相应的 Generation Agent 或上游的 Design Agent。

<strong>定向修复</strong>:借助连接器契约带来的条件独立性,路由器把现有零件划分为<strong>保留(keep)、重生成(regenerate)与新引入(newly introduced)</strong>三类子集,无故障组件留在"保留池",从而只有运动学树中受影响的节点被重新规划或重新生成。这一结构化的错误路由打破了级联失败的循环,并把冗余的 LLM 查询降到最低。

## 3 实验与结果

### 3.1 实验设置

<strong>评测基准</strong>,作者在三个基准上评估:

1. <strong>ArtiCAD-Bench(自建)</strong>:120 个装配任务,分两个子集。第一子集为 90 个多样的真实世界设计(50 个铰接、40 个静态),涵盖家具、玩具与家电,由不同模态驱动(30% 文本、30% 图像、40% 两者兼有);第二子集为从 Fusion 360 数据集筛出的 30 个工业装配(2 至 6 个零件)。
2. <strong>CADPrompt</strong>:200 项文本到 CAD 数据集,用于验证面向装配的流水线不会损害单部件生成质量。
3. <strong>ACD(Articulated Containers Dataset)</strong>:354 个物体,用于与单图铰接重建的 SOTA 方法对比,重点考察关节估计与静止态几何指标。

<strong>骨干模型</strong>:默认骨干为 Gemini-3-Flash,并额外报告 Gemini-3-Pro 的结果以展示更强骨干的效果。

<strong>Single-VLM Loop 基线</strong>:为隔离多智能体架构的贡献,作者设计了该基线——单个骨干 VLM 接收完整任务描述,在<strong>单趟(single pass)</strong>内生成一段创建并装配所有零件的 FreeCAD 脚本;它沿用相同的"生成—执行—修复"循环(至多 5 次重试),但<strong>没有</strong>设计/代码/装配的分解。基线用 GPT-5.2、Claude-Opus-4.6、Gemini-3-Flash、Gemini-3-Pro 四个骨干评测。

<strong>评测协议</strong>:由于 ArtiCAD-Bench 是无真值 CAD 模型的开放式设计,Chamfer 距离等几何指标不适用,故采用受 G-Eval、MLLM-as-a-Judge、CAD-Judge 启发的<strong>VLM 打分协议</strong>。每个装配体从多视角(铰接模型含关节运动关键帧)渲染,由 GPT-5.2、Claude-Opus-4.6、Gemini-3-Pro 三个前沿 VLM 各自独立评判,且采用思维链流程输出结构化 JSON。采用三个 1–5 李克特指标:几何(Geo.,形状准确度)、细节(Detail,特征覆盖)、运动(Motion,运动学正确性;静态项默认 5);成功率(Succ.)为二值(是否有效编译且零件齐全)。三评判者一致性稳健:Krippendorff's α 在 0.58–0.64,ICC(2,3) 超过 0.81。CADPrompt 与 ACD 则使用标准几何指标。

如[表3-1](#table3-1)所示,无论使用 Flash 还是 Pro 骨干,ArtiCAD 在几何、细节、运动与成功率上均优于所有 Single-VLM Loop 基线;其中 ArtiCAD(Gemini-3-Pro)取得全部最佳。值得注意的是,即便基线换用更强的 Claude-Opus-4.6 或 GPT-5.2,其表现也不及使用较弱 Flash 骨干的 ArtiCAD,说明增益主要来自架构而非模型容量。

## 4 应用

由于 ArtiCAD 生成的是带有类型关节与运动限位的参数化装配体,其输出可服务于超越静态 3D 内容的场景:

<strong>需求驱动的概念设计与物理样机</strong>:给定一句功能性提示(如"生成一款桌面双人玩具"),头脑风暴模块提出若干结构上不同的候选(如桌面足球、桌面冰球、桌面冰壶);对选定的"桌面足球",ArtiCAD 生成完全铰接、可直接制造的 CAD 装配体。论文配图展示了用 Bambu Lab P1S 打印并组装成可运作样机的全过程,验证了从概念设计到物理样机的贯通。

<strong>面向具身 AI 的铰接资产</strong>:流水线可把每个装配模型自动导出为带关节类型与运动限位的 <strong>URDF 文件</strong>,可直接用于 SAPIEN、Isaac Sim 等机器人仿真环境。在 Robot Viewer 中的可视化确认了导出的关节结构、轴向与运动限位被忠实保留。相比类别覆盖有限的现有铰接数据集,ArtiCAD 能按需生成新颖的、分布外的物体类型。

## 5 优势与不足

### 5.1 优势

1. <strong>填补任务空白</strong>:首个从高层描述直接生成"可编辑、铰接、带类型自由度"完整 CAD 装配体的免训练框架,介于单部件生成与不可编辑重建之间。
2. <strong>"早预测"思想巧妙且有理论支撑</strong>:用连接器契约把装配从高维拓扑匹配($O(V^{2|E|})$)坍缩为确定性坐标系对齐($O(1)$),并以马尔可夫毯/条件独立的视角论证回滚代价由 $O(K^N)$ 降到 $O(N \cdot K)$,直击 LLM 空间推理弱与拓扑命名问题(TNP)两大痛点。
3. <strong>装配阶段完全去 LLM 化</strong>:对齐为纯数学过程,带来"零幻觉装配"与可复现性。
4. <strong>免训练、可迁移</strong>:无需微调即可借经验库自演化提升;不受重建类方法的训练类别限制,可生成分布外物体。
5. <strong>工程闭环完整</strong>:从需求分析、生成、确定性装配、评审到记忆,并打通 3D 打印与 URDF 仿真,验证了实际可用性。
6. <strong>实验较为充分</strong>:三基准 + 受控基线(同骨干)+ 消融 + 多评判者一致性统计(α、ICC),论证链条相对扎实。

### 5.2 不足

1. <strong>无法表示闭环运动链</strong>:运动学树的设定排除了剪刀机构、四连杆等物理闭环结构。作者坦承这是用闭环复杂度换取确定性装配的<strong>刻意取舍</strong>,但也限制了可表达的机构类别。
2. <strong>性能上限受底层基座模型约束</strong>:作为多智能体系统,其推理与代码生成能力本质上被基座模型(如 Gemini-3、GPT-5.2)所限。
3. <strong>评测存在潜在循环性</strong>:VLM 打分协议的三个评判者(GPT-5.2、Claude-Opus-4.6、Gemini-3-Pro)与生成骨干来自同源模型族,"用 VLM 评 VLM"可能引入系统性偏好;且自建主基准缺少真值 CAD,绝对分数的客观性有讨论空间。
4. <strong>对绝对质量的天花板提升有限</strong>:消融显示经验库主要降低迭代次数而非提升上限分数;细节(Detail)分数整体偏低(约 2.9–3.1 / 5),说明精细特征覆盖仍是瓶颈。
5. <strong>开销与可扩展性未充分讨论</strong>:多智能体 + 多轮 VLM 验证 + RAG 的 token/时间成本、以及零件数大幅增加时的扩展性,论文着墨不多。

## 6 创新点

1. <strong>连接器契约(Connector Contract)</strong>:把跨零件的连接关系编码为带语义标签的局部坐标系,作为贯穿设计—生成—装配的"跨阶段契约",是全文的核心抽象。
2. <strong>设计阶段装配规划(早预测)</strong>:将装配关系预测从装配阶段前移至设计阶段,把高维拓扑匹配转化为确定性几何对齐,系统性绕开 LLM/VLM 的空间推理短板与 TNP 问题。
3. <strong>DESIGN/CODE 跨阶段回滚机制</strong>:将失败精确归因为设计级或代码级,只重调用责任智能体并做"保留/重生成/新引入"的定向修复,打破级联失败、减少冗余查询。
4. <strong>自演化经验库 + 意图驱动文档检索</strong>:以 Good/Issue 分区与非对称检索积累跨任务设计知识,并用意图驱动方式弥合"用户意图—CAD API 命名"的语义鸿沟,免微调持续改进。
5. <strong>新基准与下游闭环</strong>:提出 120 任务的 ArtiCAD-Bench,并将输出贯通到 3D 打印与 URDF 仿真,服务具身 AI 训练资产生成。

## 7 个人感悟

读完最大的体会是:<strong>这篇工作真正的"主角"不是某个智能体,而是连接器契约这一抽象</strong>。它本质上是把一个"模型不擅长"的问题(让 LLM 事后猜几何如何对齐)转化成了一个"模型擅长 + 求解器擅长"的问题(设计期声明语义连接 + 装配期数学求解)。这种"把难题搬到模型的舒适区"的思路非常值得借鉴——很多看似需要更强模型才能解决的任务,换个表示或换个阶段安排,可能用现有模型就能稳定完成。把装配阶段彻底"去 LLM 化"以换取确定性与可复现性,在工程上是相当务实且漂亮的一招。

第二点感触是,作者为"早预测优于晚预测"补上的形式化论证(搜索空间从 $O(V^{2|E|})$ 到 $O(1)$、回滚从 $O(K^N)$ 到 $O(N \cdot K)$、马尔可夫毯/条件独立)虽然在严格意义上更像是"直觉的数学化包装"而非紧致证明,但它清晰地传达了机制为何有效,也为同类多智能体/多阶段系统的"如何切分任务"提供了一个可复用的分析框架——<strong>用条件独立去隔离失败,是控制级联错误的通用利器</strong>。

也有几处让我保持谨慎。其一是评测的循环性:生成与打分共享同源模型族,加上自建主基准无真值,绝对分数需结合受控基线与定性结果一并看待,单看分数容易高估。其二是"细节"分偏低与无法表达闭环机构,提示该框架目前更适合<strong>结构清晰、以树状装配为主的日常产品</strong>,离覆盖复杂机械机构尚有距离。未来若能引入闭环约束求解、并在合成装配轨迹上做微调或强化学习(作者也指出了这一方向),这条路线的天花板会更高。总体而言,这是一篇问题定义清晰、方法洞见扎实、工程闭环完整的工作,对"如何让当前 LLM 可靠地做 3D/结构化生成"很有启发。
